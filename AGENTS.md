# AGENTS.md

## 1. 基本原则

- 只做用户明确要求的事情，不主动扩展需求
- 输出以“可执行结果”为优先，而不是解释
- 不重复用户输入内容
- 不写无意义总结、寒暄或教学式说明
- 默认读完整项目上下文后再行动

---

## 2. 代码修改规则

- 优先“最小改动”，避免大范围重构
- 不修改与当前任务无关的文件
- 保持现有代码风格（命名 / 格式 / 结构）
- 不随意引入新依赖
- 不做“顺手优化”

---

## 3. 输出格式

### 修改代码时：
- 只输出必要代码片段
- 标明文件路径
- 不重复未修改代码

示例：

```diff
# src/utils/file.ts

- const a = 1
+ const a = 2
```

## 4. 代码质量要求（强制）

- 代码必须达到生产级：
  - 高可读性（清晰命名、合理结构）
  - 高内聚、低耦合
  - 模块化设计，避免巨型函数
- 禁止“堆逻辑”式写法（屎山代码）
- 单函数职责单一（建议 ≤ 50 行）
- 避免重复代码（必要时抽取复用）

---

## 5. 注释规范（强制）

- 所有新增或修改代码必须包含**中文注释**
- 注释说明“为什么这么做”，而不是重复代码
- 关键逻辑必须解释清楚（尤其是边界处理、特殊判断）
- 不写无意义注释（如：`// 自增1`）

---

## 6. 文档同步（强制）

- 任何功能改动必须同步更新 README.md
- 文档内容包括：
  - 功能说明
  - 使用方式（如有）
  - 变更点（简要）
- 不允许代码与文档不一致

---

## 7. 变更完整性

- 修改必须是“完整可运行”的
- 不留下 TODO / 未完成逻辑
- 不提交半成品代码
- 涉及多处逻辑时必须全部同步修改，不能只改一半

---

## 8. 命名规范

- 命名必须语义清晰，禁止缩写（常见缩写除外，如 id、url）
- 变量 / 函数名必须表达真实含义
- 不使用无意义命名（如 data、temp、value）

---

## 9. 错误处理

- 所有可能失败的逻辑必须处理异常
- 不允许吞异常（catch 后无处理）
- 错误信息要明确、可定位问题
- 异步代码必须考虑失败分支

---

## 10. 边界与健壮性

- 必须考虑边界情况（空值、异常输入、极端情况）
- 不假设输入一定正确
- 对外部数据（接口 / 文件 / 用户输入）必须校验

---

## 11. 日志与调试

- 不保留调试用 console.log
- 必要日志要结构清晰、可读
- 不输出冗余日志

---

## 12. 性能意识（实际导向）

- 避免明显低效写法（如无意义循环、重复计算）
- IO / 文件 / 网络操作必须谨慎处理
- 不引入复杂优化，除非有明确必要

---

## 13. 依赖与架构

- 不随意新增依赖
- 能用原生能力解决的优先不用库
- 引入依赖必须有明确收益

---

## 14. Git 相关（如果涉及）

- 变更应具备清晰 commit 粒度
- commit message 简洁说明变更内容
- 不提交无关文件

---

## 15. 输出要求补充

- 所有改动必须：
  - 标明文件路径
  - 保证可直接应用
- 如果涉及 README 修改：
  - 同时输出 README 变更内容

## 16. 禁止过度设计

- 不为“可能的未来需求”设计复杂结构
- 不引入多余抽象层（如过早封装、过度解耦）
- 优先简单直接实现
